# GitHub Issue Labels

This document defines all the labels used for issues in the website. Consistent labeling helps in organizing, prioritizing, and tracking work efficiently.

---

## 1. Types of Issues

| Label         | Color       | Purpose                                                                      |
| ------------- | ----------- | ---------------------------------------------------------------------------- |
| `feature`     | blue        | New features or functionality                                                |
| `bug`         | red         | Bugs or unexpected behavior                                                  |
| `task`        | gray        | General tasks, non-feature work                                              |
| `enhancement` | purple      | Improvements to existing features                                            |
| `research`    | yellow      | Investigation, POC, or learning tasks                                        |
| `payment`     | med-purple  | Finance or money-related workflows                                           |
| `discussion`  | teal        | For team-wide conversations, ideas, or decisions before implementation       |
| `auth`        | stale-green | Related to user authentication, identity, and authorization workflows        |
| `security`    | gray        | Issues related to payment safety, fraud prevention, and secure data handling |

---

## 2. Priority / Status Labels

| Label             | Color      | Purpose                                     |
| ----------------- | ---------- | ------------------------------------------- |
| `high-priority`   | orange     | Needs urgent attention                      |
| `medium-priority` | gold       | Important but not urgent                    |
| `low-priority`    | light gray | Nice-to-have, low urgency                   |
| `blocked`         | dark gray  | Cannot proceed until dependency is resolved |
| `in-review`       | cyan       | PR linked, waiting for review               |

---

## 3. Optional / Contextual Labels

| Label           | Color | Purpose                     |
| --------------- | ----- | --------------------------- |
| `documentation` | green | Updates to documentation    |
| `refactor`      | pink  | Code structure improvements |
| `help-wanted`   | teal  | Task open for contribution  |
| `wontfix`       | white | Not going to implement      |

---

## Tips for Use

1. Always tag new issues with at least **one type** and **one priority/status** label.
2. Update labels as the issue progresses (e.g., add `in-review` when a PR is created).
3. Keep labels consistent — new team members should refer to this document.
